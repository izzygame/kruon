# kruon MVP PRD 调研发现

## 证据分层

- 已验证：来自项目路线图、官方资料、实际视频画面或可运行技术能力。
- 待验证：脑暴材料中的合理方向，但需要用户访谈、原型或技术 spike 证明。
- 暂不采纳：价值不足、实现成本过高，或会稀释 MVP 闭环的设想。

## 参考视频初步观察

- 画面主体是一个等距 3D/2.5D 虚拟办公室，持续叠加桌面控制面板。
- 左侧是 Agent/角色列表与邀请/编组入口；右侧是所选 Agent 的模型、状态、任务与操作区；底部是统一输入/任务指令区。
- 顶部存在多个功能导航与状态提示；中间空间通过工位、会议区、看板、灯光和角色位置表达工作状态。
- 画面呈现明暗切换、镜头平移/缩放、角色气泡与任务状态变化；视觉表达丰富，但控制面信息密度较高。
- 可转化为 MVP 的核心不是完整模拟，而是“真实任务状态 -> 可理解的空间位置/动作 + 结构化侧边栏”。
- 概念性较强、应后置验证的内容包括大规模角色数量、复杂建筑经营、深度社交模拟和纯装饰性动效。

## 项目路线图提取

- 第一性价值：可信、稳定、可观察、可干预的本地多 AI 工具控制平面；3D 是真实状态投影和传播放大器。
- MVP 闭环：15 分钟内连接；同一项目并发 2 个任务；看到真实状态；处理权限请求；收集产物；复用一次项目记忆。
- 建议接入：Codex + Claude Code 两个深适配器，外加一个通用 CLI 降级层。
- P0：工具连接、统一任务/运行模型、权限中心、2D 控制台、最小 3D 状态映射。
- P1：经过来源和作用域治理的共享记忆、双 Agent 脑暴。
- 明确不做：完整 3D 编辑器/自由建造/角色养成/实时多人；企业多租户；原始聊天全文同步；大规模插件市场。
- 统一状态至少覆盖 Idle、Planning、Running、Waiting approval、Blocked、Reviewing、Completed、Sleeping；3D 只消费稳定事件字段。
- 架构建议：Tauri 2 + React，React Three Fiber 作为可关闭的世界渲染层，SQLite + 可读文件镜像 + OS Keychain + 本地 Event Bus。
- 主要风险：适配器只能启动不能完整返回状态/权限/产物/取消时，产品会退化为漂亮启动器。
- Alpha 参考门槛：两款深适配器启动成功率 >=95%，终态一致率 >=99%，用户首周中位数完成 >=3 个可验收任务。

## 品牌约束

- 名称来源：crew + on；核心动作是 “switch your crew on.” / “一键开启你的 AI 小队”。
- 定位：面向个人创作者与小型团队的 AI 工作指挥空间。
- 品牌关键词：friendly、intelligent、connected、alive。
- 产品表达应强调“已有工具组成小队并开始协作”，避免“招聘虚拟员工”成为主叙事。

## 脑暴材料初筛

- 多份材料共同支持：本地优先、BYO 工具/订阅、共享外部记忆、任务状态/验收统一、3D/2.5D 表达工作状态。
- 可信度较高的方向：仓库/项目作为真相源；按任务风险和可验收性路由；两台设备作为生产线；适配器能力分级；2D 精确控制 + 3D 状态概览。
- 需验证而非直接采纳：共享向量记忆、全自动智能路由、群聊脑暴、跨电脑自动调度、额度自动切换。
- 暂不进入 MVP：LiteLLM 统一代理所有订阅、网页抓取/模拟登录、完整 3D 角色定制与办公经营、广告/充值入口、模型市场、皮肤商业化。
- 脑暴中的竞品、模型版本、星数、价格和“市场空白”判断均可能过时，必须通过外部一手资料复核。

### 具体甄别记录

- DeepSeek/LiteLLM 方案把 API 网关、订阅、网页工具和桌面 Agent 混为同一可路由层，技术与条款边界过度乐观；MVP 仅保留“统一任务接口 + 手动选择执行器 + 能力/健康状态”，不承诺自动接管订阅或网页端。
- 自动按额度 fallback 可能把高风险任务交给低能力执行器；MVP 应把路由建议建立在任务风险、能力声明和用户确认上，额度只作为辅助信息。
- “仓库/工作目录是真相源”“统一任务包、验收标准和交接记录”比向量记忆更适合作为首版共享上下文基础。
- 两台电脑两条生产线、分支隔离、最多两个并行任务等是特定个人工作法；可作为未来工作流模板，不应硬编码为通用产品规则。
- 3D 角色走动、会议、敲键盘、气泡与暗色夜间模式有明确表达价值；自由换装、GLB 导入、建筑经营、额度耗尽搞笑动作属于传播/付费层，不进 MVP。
- “广告极其值钱”“市场无人占据”“病毒式传播必然成立”均为未经证实的商业假设；MVP 不放广告，不让商业合作影响路由与权限。
- GPT 后半部分强调先做危险假设原型、为每个任务设置验收/测试/回滚、实现与审查分离；这些原则直接进入 MVP 的任务包与验收模型。
- “多模型脑暴”应遵循独立产出 -> 反方评审 -> 人类决策 -> 结构化结论，避免模型互相放大同一错误；首版若实现，只作为 P1 受控工作流。
- Claude 脑暴中“先用无 3D 列表界面验证控制与共享上下文，再叠加 2.5D”与路线图的控制层优先一致；PRD 将最小 3D 定义为可切换的真实状态视图，不允许阻塞核心操作。
- 中文模型生态优先是可行的差异化假设，但首发适配器仍应按可观察性、权限、取消和产物闭环选择，而不是按品牌地域数量选择。
- PRD 的任务对象应包含：目标、工作目录、允许范围、相关上下文、验收标准、测试要求、风险、回滚方式和完成报告；没有验收标准的任务不允许直接执行。
- “共享记忆”首版应收敛为可读、可版本化的项目简报/决策/约束，并由用户确认写入；向量检索和全自动提炼不是 P0。

## 外部调研 - 第一轮

- OpenAI 官方把 Codex App 定位为同时管理多个 Agent、并行运行、监督长任务的界面，说明“多任务监督”已经成为平台级基线；kruon 必须强调跨厂商和本地统一控制，而不是仅复制多 Agent 列表。
- A2A 官方规范解决跨框架/厂商 Agent 的发现、消息、任务与 Artifact 交换；MCP 更聚焦 Agent 与工具/数据源连接。两者都不能自动解决本地 CLI 的权限、取消、日志和兼容性，因此仍需适配器能力分级。
- 2026 年已出现多种 3D/2.5D AI 办公室开源项目，包括 CLAW3D、openclaw-office、agent-office、AI Agent Session Center 等；“3D 办公室”已经是拥挤形态，不能作为唯一差异化。
- openclaw-office 明确把工作状态、协作关系、工具调用和资源消耗投影到等距办公室，并配套控制台与聊天工作区；这进一步支持“双层界面”，也意味着 kruon 需以真实跨工具闭环取胜。

## 外部调研 - 协议、接入与安全

- Claude Code 官方 CLI 支持非交互 `-p`、JSON/stream-json 输出、继续/恢复会话、允许/禁止工具、权限模式、最大轮次和 hook 事件，具备深适配所需的大部分可观察接口；危险的 bypass permissions 必须在 kruon 中明确禁用或强警示。
- ACP 通过 stdio + 双向 JSON-RPC 支持一个进程上的多会话、流式 UI 更新和由 Agent 向客户端发起权限请求；它非常贴近 kruon 的本地深适配目标，可作为优先协议层。
- A2A 定义 Agent Card、带生命周期的 Task、Artifact、流式更新和异步通知，适合后续远端 Agent；MVP 本地 CLI 不应为兼容 A2A 而增加不必要服务化复杂度。
- MCP 的授权规范明确要求 token audience 绑定并禁止 token passthrough；kruon 不应把“统一凭据”实现成跨服务复用同一令牌。
- OWASP 把提示注入、工具滥用、数据外泄、记忆污染、过度自治、审批操纵、级联失败和供应链攻击列为关键风险；P0 必须包含最小权限、参数绑定审批、记忆写入确认、重试/成本上限、结构化审计和跨 Agent 不信任边界。
- Tauri 2 的 capability 系统可按窗口/WebView 限制前端暴露的系统能力，但多个 capability 会合并权限边界；架构中需让 3D/展示窗口拿不到进程、文件写入和凭据能力。

## 外部调研 - 竞品边界

- Paperclip 已经覆盖 BYO Agent、任务票据、成本预算、审批、审计和多 Agent 控制平面，并把“自主 AI 公司”作为组织隐喻；kruon 不能笼统宣称控制平面无人占据，应明确聚焦个人/小团队、本机 CLI、项目工作目录和专业 2D + 真实状态世界。
- CC Switch 已覆盖多工具配置、本地路由、failover、用量和本地密钥存储；kruon MVP 不应重造供应商路由，而应把价值放在任务、权限、产物和验收闭环。
- CLAW3D 已经提供完整 3D 办公、任务、PR、会议、跟随角色、聊天和布局编辑；openclaw-office 提供 2D 等距办公室、状态动画、协作线和气泡。由此将自由布局、会议表演和角色跟随从 kruon MVP 删除。

## 本机接入可行性核验

- 本机 Codex CLI 0.144.0-alpha.4 的非交互模式支持工作目录、sandbox、附加可写目录、JSONL 事件、结构化最终输出、会话 resume 和 review；可作为 L2 结构化 CLI 深适配起点。
- Codex 的绕过审批/沙箱和绕过 hook 信任参数被明确标记为危险；kruon 不得默认传递这些参数，并应在能力清单中标为不可自动启用。
- 本机 Claude Code 2.1.205 支持 stream-json 双向输入输出、hook 事件、会话恢复、后台 Agent、worktree、权限模式、工具 allow/deny、最大预算和结构化输出；接入能力比脑暴材料假设更完整。
- Claude Code 非交互模式会跳过工作区信任弹窗，因此 kruon 必须在自身 onboarding 中显式完成“工作目录是否可信”的确认，不能依赖上游 UI。
- 两款 CLI 都具备 P0 所需的结构化输出和受控工作目录入口，但“可靠取消、权限参数绑定、产物自动识别”仍需技术 spike 实测，不能只凭 help 文本宣称完成。

## MVP 收敛结论 v1

- 产品主张：让用户在一个可信的本地工作空间里启动、观察、干预并验收已有 AI 工具；3D 只把同一事件流变成有生命感的世界。
- 首发平台：macOS 优先；Windows 作为 Beta 之后的第二平台。理由是现有开发环境与打包复杂度，而非认为用户只在 macOS。
- 首发适配器：Codex + Claude Code 深适配；通用 CLI 仅作为实验性 L3/L4，不进入核心成功指标。
- P0 闭环：连接 -> 建立 Workspace 与策略 -> 创建带验收标准的任务 -> 选择执行器 -> 启动/流式状态 -> 权限审批 -> 暂停/取消 -> 产物/测试归档 -> 人工验收。
- P0 3D：单一低多边形微缩办公室；固定工位/评审区/审批点/交付区；6-8 个真实状态映射；点击角色打开同一 2D 运行详情；可完全关闭。
- P1：项目简报/决策/约束的提案式记忆写入与跨工具复用；双 Agent 独立研究-质疑-综合。
- 不进入 MVP：自动模型路由、网页端自动化、额度代理、跨设备同步、远端 Agent、插件市场、广告、自由建造、角色养成、实时多人、完整群体模拟。

## PRD 表达约束

- 所有功能需求必须写出触发条件、系统行为、异常/降级和可测试的验收标准。
- 所有高风险动作的审批必须绑定具体参数与有效期；审批后参数变化必须重新审批。
- 3D 状态与 2D 任务详情必须消费同一 `Run/Event` 数据，不允许各自维护状态。
- 调研结论以普通参考编号和 URL 写入文档，不携带工具内部引用标记。
