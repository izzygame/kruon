你现在不是“AI 工具太多”，而是已经凑齐了一支小型工作室，却还在用**个人游击队的方式指挥**。

真正拖效率的通常不是模型能力，而是这四件事：

1. 同一份项目上下文散落在不同账号、电脑和聊天窗口里。
2. 任务根据“谁还有额度”临时分配，而不是根据风险和能力分配。
3. 多个模型都能产出，但缺少统一的任务格式、验收和交接。
4. 创意讨论很多，却没有从“灵感”走到“证据—原型—立项”的固定漏斗。

你需要的不是再买一个神奇工具，而是一套 **AI 独立游戏工作室操作系统**。

# 一、理想形态：一脑、两手、一套外部记忆

```text
                    AA：导演 / 最终决策者
                              │
                   GPT-5.6：工作室总指挥
          规划、拆解、路由、审查、复盘，不承担全部执行
                              │
                Git 仓库 / GitHub：唯一真相源
             项目状态、任务、决策、规范、交接全部在这里
                    ┌─────────┴─────────┐
                    │                   │
             Mac：主程与集成线      Windows：并行生产线
             Codex Pro 5.6         Codex Plus / CodingPlan
             高风险、跨模块         独立功能、实验、测试
                    │                   │
                    └──── PR / 测试 / 审查 ────┘
                              │
               Workbuddy / Qoderwork / Hermes
                    小需求、低风险、应急产能
```

关键不是让 GPT 真的远程操作所有网页和账号，而是让所有工具都接收同一种“任务包”，并把结果交回同一个地方。

一句话概括：

> **模型不是记忆，仓库才是记忆；聊天不是管理，任务状态才是管理。**

GPT-5.6 可以当总指挥，但它不能成为唯一信息载体。否则换账号、换电脑、开新对话，工作室就失忆了。

# 二、先建立“工作室控制层”

建议在《此生修仙》的仓库里增加一个 `.studio` 目录：

```text
.studio/
├── STUDIO_STATE.md       # 项目现在处于什么阶段
├── CURRENT_SPRINT.md     # 当前只做什么，暂时不做什么
├── TASK_QUEUE.md         # 待分配任务，按风险分层
├── ROUTER.md             # 哪类任务交给哪个工具
├── DECISIONS.md          # 重要设计和技术决策
├── BRIEFING.md           # 每次开工给总指挥看的压缩上下文
├── SCORECARD.csv         # 各工具实际表现记录
├── handoffs/             # AI 任务交接记录
└── prompts/              # 稳定使用的任务模板
```

游戏项目再配几份稳定规范：

```text
docs/
├── VISION.md             # 游戏愿景、一句话卖点、玩家幻想
├── DESIGN_PILLARS.md     # 3～5 个不可动摇的设计支柱
├── SYSTEM_MAP.md         # 系统、模块和依赖关系
├── ART_BIBLE.md          # 美术规范
├── VOICE_BIBLE.md        # 文案口吻、禁用表达、示例
├── AUDIO_BIBLE.md        # 音乐氛围和使用规则
└── ASSET_LICENSES.csv    # 音乐、美术、字体、插件授权记录
```

其中最重要的是 `BRIEFING.md`。它不是详细设计文档，而是一页“战情简报”，只保留：

* 当前版本和阶段。
* 最近完成了什么。
* 目前最重要的三个问题。
* 当前分支、测试状态和已知缺陷。
* 接下来唯一主目标。
* 哪些决策已经冻结，禁止模型擅自推翻。

以后无论 Codex、Claude Code、Workbuddy 还是 Hermes，都先拿到这份简报，再拿具体任务包。这样两个账号、两台电脑之间不会不断重复解释背景。

# 三、不要按“额度”路由，要按“任务风险”路由

你目前已经隐约形成了分层，但还可以把它制度化。

给每个编程任务评五项，每项 0～2 分：

| 维度    | 0 分     | 1 分    | 2 分         |
| ----- | ------- | ------ | ----------- |
| 需求模糊度 | 完全明确    | 有少量判断  | 需要重新设计      |
| 跨模块范围 | 单文件/单模块 | 2～3 模块 | 全局影响        |
| 回归风险  | 容易回滚    | 中等     | 存档、核心循环等高风险 |
| 上下文需求 | 很少      | 需要相关文档 | 需要理解全项目     |
| 决策重要性 | 临时小改    | 正式功能   | 架构或产品级决策    |

总分决定分配：

| 得分   | 任务类别              | 默认工具                              |
| ---- | ----------------- | --------------------------------- |
| 0～3  | 低风险、边界清楚          | Workbuddy、Qoderwork、Hermes        |
| 4～6  | 中等规模、可独立验收        | Codex Plus、火山 CodingPlan、可用时的 GLM |
| 7～10 | 高风险、复杂、跨模块        | Codex Pro + GPT-5.6               |
| 特殊覆盖 | 存档、经济系统、核心战斗、架构迁移 | 无论分数多少，交给主力线                      |

最重要的限制是：

> **便宜模型只能接“容易描述、容易验收、容易回滚”的任务。**

不能因为 5.6 没额度，就把一个模糊的核心系统改造扔给便宜模型。那通常不是省钱，而是把你自己变成免费返工员工。

# 四、给每个工具一个固定岗位

根据你自己的实际体验，我建议这样编制。

| 工具                                   | 默认岗位                    | 不建议承担             |
| ------------------------------------ | ----------------------- | ----------------- |
| Codex Pro / GPT-5.6                  | 主程、架构、复杂需求、任务拆解、最终集成    | 零碎格式修改、小文档        |
| Codex Plus                           | 第二主程、中高复杂独立模块、代码审查      | 与 Pro 同时修改同一系统    |
| 火山 CodingPlan + Claude Code          | Codex 替补、边界明确的中型功能      | 在缺乏设计文档时重构全局      |
| GLM 5.2                              | 高价值替补、复杂问题第二意见          | 用稀缺额度修小拼写错误       |
| Workbuddy                            | 小功能、小 Bug、脚手架、重复性修改     | 存档、架构、核心玩法        |
| Qoderwork                            | 与 Workbuddy 竞争同一低中风险任务池 | 无验收标准的开放式需求       |
| Hermes + DeepSeek                    | 应急、低成本修复、事件文案初稿、批量变体    | 直接决定游戏核心设计        |
| GPT / Claude / Gemini / DeepSeek 网页端 | 独立研究、创意发散、反方审查          | 一群模型互相吹捧后直接立项     |
| GPT / Gemini 生图                      | 构图探索、草图、占位图、风格试验        | 尚未锁定风格时批量生产正式资源   |
| 即梦 / Midjourney                      | 风格锁定后的生产级资源候选           | 一上来两家同时订阅、同时大批量生产 |

Workbuddy 和 Qoderwork 不需要凭感觉轮换。让它们竞争，用 `SCORECARD.csv` 记录：

```text
任务类型
使用工具
准备任务说明耗时
第一次产出是否通过
返工次数
你人工修改耗时
最终是否采用
费用或额度消耗
```

真正应比较的不是“单次调用有多便宜”，而是：

> **一个可接受结果的总成本 = 调用成本 + 描述成本 + 审查成本 + 返工成本。**

# 五、两台电脑不要当两套工作环境，要当两条生产线

按照你现在 Pro 在 Mac、Plus 在 Windows 的配置：

## Mac：主程与集成线

Mac 负责：

* Codex Pro 的复杂开发。
* 架构和核心系统。
* 主分支集成。
* 重要代码审查。
* 版本决策和状态更新。
* 最终合并。

## Windows：并行生产与验证线

Windows 负责：

* Codex Plus 或 CodingPlan 的独立功能。
* 小 Bug 批处理。
* Windows 构建和兼容性测试。
* 数据校验、编辑器工具。
* 内容生成管线。
* 实验性分支和原型。

执行规则：

1. 两台电脑永远不要直接修改同一个分支。
2. 每个任务对应一个 Issue 和一个分支，例如：
   `feat/CSXX-142-cultivation-event-filter`
3. 同一时间最多两个编程任务并行。
4. 两个并行任务不得碰同一个核心子系统。
5. Windows 产出通过提交或 PR 交给 Mac，不通过聊天窗口交代码片段。
6. 高风险任务由不同模型复审，尽量不要让实现者自己宣布“审查通过”。

并行度并不是越高越好。独立开发中，第三个同时开发的 AI 往往不是增加产能，而是在增加合并税。

# 六、建立“额度模式”，额度不足时切换工作，不是硬换模型

你可以设置三个工作模式。

## 绿色模式：5.6 额度充足

优先处理：

* 复杂架构。
* 核心玩法。
* 跨模块功能。
* 顽固 Bug。
* 重构和技术债决策。
* 关键任务拆解。

## 黄色模式：只剩 Plus、CodingPlan 或 GLM 替补

处理：

* 已经拆清楚的独立功能。
* 单系统增强。
* UI、编辑器工具。
* 测试和性能分析。
* 中等复杂度 Bug。

## 红色模式：只剩免费或低成本模型

不要硬啃复杂需求，转向：

* Bug 复现和最小复现案例。
* 单元测试、回归测试。
* 数据清洗和校验器。
* 文档、注释、重命名。
* 事件文案初稿和批量变体。
* 美术资产清单。
* 授权信息整理。
* 新项目研究。
* 下一批高风险任务的规格编写。

这样即使 Codex 额度耗尽，整个工作室仍然有高价值工作可做，而不是停摆或让弱模型破坏主干。

# 七、你缺失的“创意—研究—立项”流程

多模型脑暴不能直接等于创意流程。正确顺序应该是：

```text
外部信号 → 机会假设 → 独立发散 → 反方攻击
→ 低成本原型 → 玩家/自身体验证据 → 立项
```

## 第一阶段：机会采矿

建立一个单独的 `studio-lab` 仓库，或者在当前仓库增加：

```text
ideas/
├── inbox/          # 一句话灵感、玩家抱怨、机制碎片
├── research/       # 已经开始研究的方向
├── concepts/       # 成型的项目卡
├── prototypes/     # 做过验证的项目
├── killed/         # 被否决，但保留原因
└── greenlit/       # 正式立项
```

机会来源不要只看“什么类型最近火”，而要收集四类信号：

* 玩家反复表达但没有被满足的欲望。
* 某个游戏中大家喜欢，但原作没有继续深入的机制。
* 你自己长期着迷的主题。
* 由于 AI、程序生成或现有技术，现在突然变得可低成本实现的体验。

可以用这个公式判断是否值得继续研究：

> **项目机会 = 强烈玩家欲望 × 可传播钩子 × 你的产能优势 × 可低成本验证性**

## 第二阶段：多模型独立发散

GPT、Claude、Gemini、DeepSeek 不要一开始互相看答案。

让它们拿同一份研究材料，各自独立产出：

* 机会解释。
* 目标玩家。
* 12 个概念。
* 每个概念的一句话钩子。
* 最大风险。
* 最小验证方法。

之后再由 GPT-5.6 汇总。

不先独立思考就开圆桌，最后很容易变成四个模型共同把一个普通点子夸成宇宙级创意。

## 第三阶段：项目概念卡

每个进入候选区的项目必须填完：

```text
项目代号：
一句话介绍：
玩家扮演谁：
玩家最核心的幻想：
10 秒内能展示的钩子：
核心循环：
三个设计支柱：
与已有游戏的关键差异：
目标玩家：
最小可玩原型是什么：
不做哪些内容：
最大技术风险：
最大内容风险：
最依赖美术的地方：
哪些环节能被 AI 放大：
立项前必须证明什么：
什么结果出现时立刻终止：
```

“终止条件”非常重要。没有终止条件的项目研究，最后都会变成舍不得扔的半成品。

## 第四阶段：反方评审

让模型扮演不同角色，分别攻击项目：

* **玩家代表**：为什么玩十分钟就关掉？
* **制作人**：哪里会超范围？
* **市场人员**：商店页第一张图表达得清楚吗？
* **技术负责人**：最可能延期的系统是什么？
* **内容负责人**：是不是需要无限喂内容才能维持？
* **悲观投资人**：为什么这个项目不该做？

立项前，批评比鼓励更值钱。

## 第五阶段：原型证明

立项不是靠模型评分，而是靠原型证据。

先验证最危险的假设，例如：

* 核心操作到底有没有乐趣。
* 选择是否真的有张力。
* 玩家是否理解目标。
* 10～20 秒的视频是否能传达特色。
* 不依赖最终美术时，玩法是否仍然成立。
* 内容生产量是否在你的能力范围内。

如果核心乐趣必须等几十张正式美术、完整剧情和大量数值全部到位才看得出来，这本身就是高风险信号。

## 第六阶段：正式立项评分

可以使用下面的权重：

| 项目          | 权重 |
| ----------- | -: |
| 灰盒原型的实际乐趣   | 25 |
| 一句话和短视频传播力  | 20 |
| 与个人产能的适配    | 20 |
| 差异化         | 15 |
| 可观察到的玩家需求   | 10 |
| 你本人是否愿意长期投入 | 10 |

立项需要同时过四道门：

> **我想做、我能做、玩家看得懂、最危险假设已经被验证。**

# 八、立项后怎样进入超高效率生产

正确顺序不是“设计完成后开始批量写功能”，而是：

```text
愿景冻结
→ 核心玩法原型
→ 垂直切片
→ 内容和资产生产管线
→ 批量生产
→ 打磨与发布
```

## 1. 先做垂直切片，不先铺开内容

垂直切片应该包含一小段接近正式质量的完整体验：

* 核心玩法。
* UI。
* 一段内容。
* 美术方向。
* 音频方向。
* 存档或必要状态。
* 一次完整玩家循环。

它的目标不是“做了多少”，而是证明整个生产链能闭环。

## 2. 把需求拆成 AI 可执行任务

低成本工具最需要严格的任务包：

```text
任务 ID：
任务目标：
为什么要做：
当前行为：
期望行为：
允许修改的文件或模块：
相关接口和约束：
明确不做什么：
验收条件：
需要运行的测试：
潜在风险：
失败后如何回滚：
完成后必须汇报：
```

最后一项固定要求 AI 回报：

```text
1. 修改了哪些文件
2. 为什么这样实现
3. 跑了哪些测试
4. 仍有哪些已知问题
5. 是否引入新的依赖或配置
6. 下一位接手者需要知道什么
```

这样 Workbuddy、Qoderwork、Hermes 才能真正成为工人，而不是随机发挥的实习生。

## 3. 让不同模型实现和审查

推荐组合：

* Codex Plus 实现，GPT-5.6 审查。
* CodingPlan 实现，Codex Pro 集成。
* Workbuddy 修复，Qoderwork 或 Hermes 做针对性复查。
* DeepSeek 写事件初稿，GPT 按 `VOICE_BIBLE.md` 做统一编辑。

重点不是追求“两个模型意见一致”，而是让审查模型主动找：

* 未覆盖边界。
* 需求误解。
* 隐式状态变化。
* 存档兼容。
* 性能回退。
* 测试缺口。
* 与现有设计支柱冲突的地方。

# 九、美术和内容生产不要过早扩张

你对即梦和 Midjourney 的判断很合理，但当前不建议直接双订阅并开始全量替换。

先做一份 10 张的“生产基准包”，覆盖：

* 主角或关键角色。
* NPC。
* 场景。
* 图标。
* 道具。
* UI 装饰。
* 宣传图。
* 不同动作或表情。
* 复杂构图。
* 同角色一致性。

给 GPT、Gemini、即梦、Midjourney 使用完全相同的需求和参考，再评估：

* 第一次可用率。
* 角色一致性。
* 风格一致性。
* 修改成本。
* 后期处理时间。
* 商用生产速度。

最终先选一家作为主生产线，另一家只在明确短板处补位。否则很容易拥有两套都不错、但互相不统一的资源。

正式批量生产前必须锁定 `ART_BIBLE.md`：

* 色彩和明度范围。
* 光照。
* 线条或材质特征。
* 人物比例。
* 构图规则。
* 镜头角度。
* 背景复杂度。
* 禁止出现的元素。
* 正面和反面示例。
* 后期处理步骤。
* 文件尺寸和命名规范。

文案同理。Hermes/DeepSeek 可以承担事件初稿，但最终必须通过统一的 `VOICE_BIBLE.md`，否则不同模型写出来会像游戏里住着五个时代的编剧。

# 十、现在暂时不要急着再买 OpenCode

按照你当前的配置，Codex Pro、Codex Plus、火山 CodingPlan，再加免费工具，已经具备完整的三级产能。

OpenCode 是否值得买，应该由数据触发，而不是由“万一额度又不够”的焦虑触发。满足下面情况再考虑：

* CodingPlan 经常成为实际瓶颈。
* 有足够多适合中高复杂替补模型的任务。
* 当前替补的返工成本显著偏高。
* 新订阅能明显降低“可接受结果总成本”。
* 你已经有统一任务包，新工具可以即插即用。

否则它只会成为新的聊天窗口、新的配置、新的上下文孤岛。

# 十一、可直接使用的“总指挥提示词”

把下面内容放进一个专门的 GPT-5.6 项目，长期只用于工作室调度：

```text
你是我的独立游戏工作室总指挥。
```

# 十二、最先落地的三件事

1. **先在《此生修仙》仓库建立 `.studio` 目录和六个核心状态文件。**
2. **把现有待办按高、中、低风险重新排队，而不是继续按额度临时找模型。**
3. **固定 Mac 为主程集成线、Windows 为并行生产线，从下一批任务开始只通过任务包和提交交接。**

先用《此生修仙》把这套系统跑顺。等协议稳定后，再考虑用脚本、MCP 或本地调度器自动生成简报、领取 Issue、创建分支和汇总交接。不要先造一个庞大的多智能体平台——自动化一个混乱流程，只会更快地产生混乱。
