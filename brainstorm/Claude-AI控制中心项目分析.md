# 本地 AI 控制中心 + 游戏化可视化界面 —— 项目分析

> 基于 AA 自身多工具、多订阅的真实使用痛点，结合当前（2026年7月）行业格局进行的前期思考文档。

---

## 一、先说结论：这个方向是对的，但赛道已经不是空的

在动手之前必须先纠正一个前提判断：这不是一条无人区赛道，而是一条**正在快速拥挤、但尚未有赢家**的赛道。这个判断会直接决定后面的定位怎么写。

搜索目前的市场后发现，"游戏化 3D/2.5D 办公室 + AI 团队管理"这个界面形态，已经有多个项目在做，而且你截图里那款、以及你提到的 OpenClaw，本身就属于这个类别：

- **OpenClaw / CLAW3D**：开源的 3D 虚拟办公室，可以看 agent 开会、走动、处理任务，支持锁定视角看某个 agent 实时工作，甚至能拖家具布置办公室。这和你截图里的产品几乎是同一形态。
- **Cozy Office**：主打全云端架构 + 2.5D 办公室，卖点是"招聘 AI 员工、分配任务、追踪进度、接第三方服务"一站式完成。
- **TaskVille**：虚拟办公室，AI 团队按计划自主执行任务，用"能量/钻石"做游戏化资源系统，招募和分配 agent 变成一种策略性消耗。
- **SimOffice**：目前看起来最接近"消费级 SaaS"的一款，211+ 预制 agent、29 个第三方集成，办公室里 agent 走动协作，定价 29.99 美元/月，支持接入 Claude/GPT/Gemini/Grok。
- **Pixel Agents / AgentOffice**：开源、像素风，桌子即目录、agent 有血条一样的 token/context 状态展示，愿景写得很直白——把管理 AI agent 做成"玩模拟人生"的体验，工具无关、可扩展任意 AI CLI。

这些项目普遍验证了两件事：第一，"游戏化可视化"确实是能打的界面形态，用户认知成本低、传播性强；第二，目前几乎所有做得成型的项目，卖点都落在**"帮你招募/托管一批 AI 员工去干活"**，也就是自建或接入 agent 市场，做的是"人力资源管理"的比喻。

而你的核心洞察——**"我已经有一堆正在付费的 CLI/订阅/API，我要的不是新招一批 agent，是把已经在用的这些整理、调度、共享记忆"**——目前在上面这些产品里都没有被真正满足。这才是你和它们的分野，而不是"有没有 3D 界面"。

另外，工具层还有一类完全不同的对标：**cc-switch**。这个项目验证了你提到的"小工具撬动 API 中转商广告主"模式确实跑通了——开源、跨平台，统一管理 Claude Code、Codex、OpenCode、Gemini CLI、OpenClaw 等 CLI 工具的供应商配置、MCP、Skills、用量统计，GitHub 星数已经很高，页面上挂着多个 API 中转平台的赞助位（RunAPI、APIKEY.FUN、TeamoRouter、CCSub 等），这些中转商通过赞助/专属注册链接获客，这正是你设想的商业模式的现实先例。但 cc-switch 本身没有可视化、没有协作、没有共享记忆，它只解决"切配置"这一件事，做得足够深但也足够窄。

**一句话概括当前格局**：游戏化可视化的公司在做"agent 招聘市场"，cc-switch 这类工具在做"配置管理"，企业级的 GitHub Copilot App / Augment Intent / Guild.ai 在做"生产级多 agent 协作与治理"。三者中间，"个人开发者的多订阅本地编排 + 共享记忆 + 游戏化界面"这个交集目前没人占住。

## 二、真正的差异化定位

基于上面的格局，建议把定位收窄到这句话：

> **不是又一个 agent 招聘市场，而是你已有工具、已有订阅、已有 CLI 的"本地指挥部"。**

具体体现在三个"不做"和三个"做"：

**不做**：
1. 不自建/托管 agent，不做 agent 市场（这是 SimOffice/TaskVille/Cozy Office 在做的事，需要承担模型调用成本，重资产）
2. 不做云端托管（避免和 Cozy Office、SimOffice 在"谁的云更稳"上竞争，也规避你最在意的隐私/账号安全问题）
3. 不重新发明配置管理（cc-switch 已经做得很好，可以考虑集成而不是重造）

**做**：
1. **BYO 一切**：用户已有的 Codex、Claude Code、GLM、Qwen、DeepSeek、Hermes 等，全部作为"可插拔的执行体"接入，工具本身不产生模型调用成本
2. **跨工具共享记忆**：这是目前所有同类竞品都没做深的点——多个割裂的 CLI/网页会话之间，项目上下文、决策历史、设定圣典能够共用
3. **中文模型生态优先**：混元、GLM、Qwen、DeepSeek、智谱，这些在 SimOffice/Cozy Office 这类欧美产品里基本不支持或支持很浅，而这恰恰是国内个人开发者的真实生态

这个定位的额外好处：因为不托管模型调用，你的服务器成本结构会非常轻——本地优先架构下，重计算都在用户自己的订阅账号里跑，你的后端只需要承担同步、记忆索引、可能的社区/工作流市场，这和"体量可控"的诉求是一致的。

## 三、核心功能模块

### 1. 工具接入层（Adapter Layer）
- 每个外部工具（Codex CLI、Claude Code、GLM 网页/API、Qwen、DeepSeek API、Hermes、网页端模型）对应一个适配器，负责：拉起进程/发起请求、读取输出、状态上报（空闲/工作中/需要确认/报错）
- 参考 Pixel Agents 的做法：定义统一的 HookProvider 接口，新增一个工具只需要新写一个子目录的适配代码，不动核心
- 优先支持能通过 CLI/本地进程直接交互的工具（Codex、Claude Code、OpenCode 类），网页端模型（Gemini/智谱网页版）可以先做"半自动"——弹出待办、用户手动去对应网页执行后回填结果

### 2. 共享记忆层
- 项目级的"中枢文档"结构化存储（设定圣典、决策日志、任务队列），本地 Markdown/SQLite 存储，所有适配器在下发任务前自动注入相关上下文
- 参考 agent-office 系列的做法：本地向量库（可选，非必须）+ 结构化 Memory Store，避免每次都要人工去拼上下文

### 3. 调度与路由引擎
- 把你现在"复杂需求上 Codex、小 bug 上 GLM/Qwen、额度告急上 DeepSeek"的经验，变成可配置的路由规则（任务类型 → 优先工具 → 备选链）
- 额度/账号健康看板：每个订阅/API 的余量、重置时间、当前状态，避免创作高峰期几个额度同时见底

### 4. 群聊脑暴模式
- 多模型同一话题并行提问，产出强制走向一份结构化 spec（问题定义、核心结论、待验证点），而不是停留在聊天记录里
- 可以设定角色（比如"文化真实性审查""机制可行性""挑刺"），对应分配给不同模型

### 5. 可视化游戏化界面
- 2.5D/3D 办公室隐喻：每个接入的工具/CLI 是一个"角色"，状态（空闲/工作中/等待确认/报错）用动作和气泡展示
- 桌子/工位 = 项目或工作目录，拖拽切换 agent 负责的项目，参考 Pixel Agents 的"桌子即目录"设计
- Token/额度状态做成血条式展示，这个在 Pixel Agents 里已经验证是好用的隐喻，可以直接借鉴思路（不是抄界面，是抄这个信息可视化的逻辑）

## 四、技术架构建议（早期，够用即可）

- **客户端**：Tauri（Rust + Web 前端）而不是 Electron——cc-switch 就是 Tauri 2 + Rust + React 的组合，已经验证在这类"本地进程管理 + 轻量 GUI"场景下体积小、启动快、内存占用低，值得直接参考它的技术选型
- **可视化层**：早期用 2.5D（Phaser.js 这类 2D 游戏引擎）而不是真 3D——agent-office/AgentOffice/Pixel Agents 这几个开源项目都验证了 Phaser.js 路线工程量可控、效果也足够"游戏感"，真 3D（Three.js/Unity）留到验证完产品逻辑之后再做视觉升级，避免第一步就把工程体量做重
- **本地存储**：SQLite 存配置和记忆结构化数据，Markdown 文件存中枢文档（方便用户自己也能直接编辑、纳入 git 版本管理，这点对你这种本身就在用 AGENTS.md/CLAUDE.md 约定的开发者尤其友好）
- **进程编排**：本地起子进程/调用本地 CLI，避免把用户的 API Key 上传到你的服务器（安全和信任是这类工具的生命线，cc-switch 的口碑很大程度建立在"配置本地存储、原子写入"这种细节上）

## 五、商业化：广告为主，但需要补充结构

cc-switch 的先例证明"开源工具 + API 中转商赞助位"这个模式确实能跑，但要注意它的天花板和风险：

- **可行**：工具本身免费/开源，在"添加供应商"环节植入中转商的预设模板和推广位，中转商按注册数或充值分成付费，这是你提到的模式，逻辑成立，而且已经有多个中转商在抢这个位置（RunAPI、APIKEY.FUN、TeamoRouter、CCSub、Unity2.ai 等），说明这个渠道的采购意愿是真实存在的
- **风险**：这类广告位的价值高度依赖工具本身的用户规模和活跃度，早期（几千到几万用户量级前）议价能力很弱，广告收入很可能覆盖不了哪怕是最基本的运营成本
- **建议补充的变现点**（不冲突，可以叠加）：
  - 工作流/角色模板市场（比如"小说创作四人协作流""游戏开发五人协作流"），社区贡献 + 官方抽成，这和你自己"立项脑暴协议"的思路是一脉相承的，天然有内容沉淀
  - 可视化皮肤/角色形象的付费定制——SimOffice、Cozy Office 这类产品的游戏化外观本身就有消费吸引力，皮肤类内购摩擦小、边际成本几乎为零
  - 云同步/多设备协同的可选增值服务（你自己就有两台电脑的真实需求，这个痛点具备一定普遍性）

## 六、主要风险清单

1. **适配器维护成本**：每个上游工具（Codex、Claude Code、GLM 官方、Qwen、DeepSeek……）的接口、CLI 参数、认证方式都会不定期变化，你是在给一堆"别人家的产品"做外挂式集成，对方一次更新可能让你的适配器失效，这是长期最大的隐性工程成本
2. **账号安全与合规边界**：管理用户多个平台的 API Key/登录态，哪怕全本地存储，也要在协议里说清楚责任边界；如果涉及自动化操作网页端模型（模拟登录/操作 DOM），要格外小心触碰对方的服务条款
3. **同赛道拥挤度**：如上所述，"游戏化 AI 团队管理"已经有至少五六个项目在做，且不乏融资/较高关注度的项目（SimOffice、TaskVille），单纯"界面好看"不构成壁垒，壁垒必须落在"跨工具共享记忆"和"BYO 订阅路由"这两个差异点上，做深不做全
4. **纯广告模式早期现金流风险**：如商业化部分所述，需要有除广告外的第二增长点做缓冲
5. **一人/小团队工程节奏**：适配器数量、可视化打磨、记忆系统三线并行如果同时铺开容易失焦，建议 MVP 阶段只做一条主线打透

## 七、MVP 建议范围

**第一阶段（验证核心假设：共享记忆 + 路由是否真的比现状好用）**
- 只接入 2-3 个你自己每天在用的工具（比如 Codex CLI + Claude Code + 一个网页端模型的半自动接入）
- 中枢文档（Markdown）+ 简单路由规则，先不做可视化，用列表式界面验证"共享上下文"这件事本身的价值
- 自己作为唯一用户跑满一个完整项目周期（比如这一世修行的一次版本迭代），验证是否真的减少了"重复交代上下文"的痛苦

**第二阶段（可视化 + 群聊脑暴）**
- 上 2.5D 可视化层（Phaser.js 路线），把第一阶段的路由和状态可视化
- 加入群聊脑暴模式和强制产出 spec 的机制

**第三阶段（开放接入 + 早期用户）**
- 适配器接口标准化、开放给小范围用户自测，重点收集"新工具接入难度"和"记忆共享是否真的省心"两类反馈
- 开始接触 API 中转商谈赞助位，此时你已经有真实活跃用户数据，议价空间更大

## 八、需要你来定的关键决策

1. **优先平台**：先做 macOS 还是 Windows，还是两端同步做？（你自己的双机场景决定了这不是小问题，SimOffice 目前也还只支持 macOS）
2. **半自动 vs 全自动**：网页端模型（Gemini/智谱网页版）短期内是接受"半自动回填"，还是投入工程去做浏览器自动化？
3. **要不要做开源**：cc-switch 的口碑和信任感很大程度来自开源可审计，这和你"广告位模式"直接相关——开源是否也是你计划里的一部分？
4. **第一批适配器选谁**：建议就是你自己每天用的那几个（Codex、Claude Code、GLM/Qwen 二选一），而不是一开始就求全
