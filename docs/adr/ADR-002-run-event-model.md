# ADR-002：Run 投影与追加式 Event 日志

- 状态：已采纳
- 日期：2026-07-14

## 背景

产品需要同时展示当前运行状态、保留协议证据，并在应用重启后恢复或明确降级。

## 决策

SQLite 保存 `runs` 投影、`events` 追加日志和 `schema_migrations`。事件以 `(run_id, sequence)` 连续保序，`event_id` 全局唯一；完全一致的重复事件幂等，不一致重复、跳号、倒退和终态后追加均冲突。事件与 Run 投影在同一事务更新。

非终态 Run 在重启时不操作历史 PID，而是追加恢复事件并进入 `uncertain`。同一 Run 只允许一个最终终态事件。

## 后果

- `get_run` 快速读取投影，`replay_run` 可从事件验证同一终态。
- 异常退出不会误杀 PID 已复用的无关进程。
- W1 用单个 `Mutex<Connection>` 保守串行化写入；这是吞吐权衡，不是长期连接池方案。
- 损坏、冲突或不连续数据以失败关闭，不自动猜测修复。

## 未选择方案

- 只保存可变 Run 行：无法审计协议和恢复过程。
- 文件日志：并发、事务和查询保证不足。
- W1 即引入事件总线/远端数据库：超出本地闭环范围。

## 复审条件

并发写入成为可测瓶颈，或需要多进程访问同一数据库时，引入连接池/单写者队列并保持当前事件契约。
