# kruon MVP 开发计划

版本：v0.1  
日期：2026-07-11  
依据：`kruon_MVP产品需求说明书_2026-07-11.docx`（已审查通过）

## 1. 交付目标

在 10 周内交付 macOS 邀请制 Alpha，使目标用户能够在一个本地可信工作空间中：

1. 发现并连接 Codex CLI 与 Claude Code CLI；
2. 创建带范围、风险与验收标准的任务；
3. 并行启动、观察、审批、取消和恢复两个 Run；
4. 集中查看文件、diff、测试和完成报告；
5. 对任务执行人工验收或退回；
6. 在不损失任何控制能力的前提下，可选使用 3D 世界视图理解状态。

本计划不包含 P1 项目记忆、自动路由、远端 Agent、云同步、插件市场、网页自动化、复杂 3D 经营和多人协作。

### 1.1 定位、部署与凭据边界

- 产品不要求用户登录 Kruon；接入 Codex、Claude Code 或其他云端工具时，用户仍需满足该工具自身的认证与网络要求。
- Workspace、事件记录和产品状态默认保留在本机。MVP 不建设隐式云同步、遥测回传或集中式凭据服务。
- 本地模型是后续产品方向，而非 W1/MVP 已交付能力。只有在模型进程、Provider 端点和配套工作流均被验证为本地后，UI 才可标记“离线”。
- Provider API key 不得进入项目文件、SQLite、事件、日志或诊断包。后续凭据能力使用 OS 安全存储、Provider 级显式授权与最小范围临时注入，详见 ADR-006。
- 代码公开不自动构成开源许可；在对外发布“开源”主张前必须确定许可证并提交 `LICENSE` 文件。

## 2. 排期假设

- 1 名产品型客户端/全栈工程师：Tauri、React、SQLite、领域模型、2D 控制面与打包。
- 1 名系统/适配器工程师：CLI 进程、结构化协议、权限、取消、恢复、兼容 fixtures。
- 0.5 名 3D/交互工程师：W7 起投入 React Three Fiber 世界视图。
- 产品与设计由创始人主导；W1-W2 每周至少完成 3-5 名目标用户访谈。
- Alpha 前安排一次外部安全评审。

若只有 1 名全职工程师，建议保留相同阶段顺序，把目标周期调整为 16-18 周；不得削减权限、取消、终态一致与人工验收门槛来换取 10 周表面完成。

## 3. 技术基线

### 3.1 推荐架构

- 桌面壳：Tauri 2。
- 前端：React + TypeScript；2D 控制面作为默认主界面。
- 3D：React Three Fiber，独立懒加载并可完全卸载。
- 本地核心：Rust；负责进程治理、策略校验、文件边界、审计与 IPC。
- 持久化：SQLite；事件追加写入，派生当前状态。
- 凭据：macOS Keychain；不复制或透传上游 token。
- 本地模型与 Provider 凭据：后续独立适配器/设置能力；不通过全局环境变量继承或项目配置文件实现。
- 事件：单一规范化 Event 流；2D、3D、通知和诊断只消费这一来源。
- 适配器：`Adapter` 契约 + Codex/Claude 两个深适配实现；版本能力清单独立管理。

### 3.2 模块边界

| 模块 | 核心职责 | 禁止承担 |
|---|---|---|
| Domain Core | Workspace、Policy、Task、Run、Event、Approval、Artifact 状态机 | CLI 文本解析、UI 状态 |
| Policy Engine | 文件、命令、网络、预算与审批决策 | 直接执行动作 |
| Process Supervisor | 启动、信号、进程树、超时、退出与恢复 | 判断业务完成 |
| Adapter Host | 适配器版本探测、输入输出、能力降级 | 绕过 Domain/Policy 写库 |
| Event Store | 追加事件、快照、迁移、崩溃恢复与对账 | 保存上游凭据 |
| Artifact Service | 文件、diff、测试、报告识别与验收归档 | 自动判定用户接受 |
| Desktop UI | 任务、审批、交付、连接与设置 | 直接访问系统进程/凭据 |
| World View | 把规范事件映射为固定空间状态 | 执行、审批或维护独立状态 |

### 3.3 首批架构决策记录（ADR）

- ADR-001：为什么选择 Tauri 2，而不是 Electron。
- ADR-002：事件追加 + 派生状态，而不是只存当前状态。
- ADR-003：固定双适配器协议与进程组治理。
- ADR-004：工作区、环境、存储与 IPC 安全边界。
- ADR-005：Codex 只读、临时、参数受控集成。
- ADR-006：local-first、凭据保管与本地模型离线语义。
- ADR-007（已采纳）：2D 与 3D 同源事件及 3D 能力隔离。

### 3.4 已识别的关键技术不对称

本机复核确认：顶层交互式 Codex 支持 `--ask-for-approval`，但 `codex exec` 非交互模式没有逐动作审批参数；Claude Code 则提供 stream-json、manual permission mode 与 hook 事件入口。因此不能预设两款深适配器具有完全相同的审批能力。

W1 必须完成以下决策：

1. spike Codex 实验性 `app-server` / `exec-server` 是否暴露稳定的逐动作审批通道；
2. 若不成立，评估 PTY 交互适配的可靠性与维护成本；
3. 若两者均不满足 Alpha 质量门，Codex 适配器必须明确降级为 `sandbox_policy_only`，通过前置策略限制高风险动作，并在 UI 中说明与 Claude 逐动作审批的差异。

该问题是 W2 进入 M1 前的决策门，不允许延后到审批 UI 开发阶段。

## 4. 关键路径与并行泳道

### 4.1 关键路径

`适配器 spike → 统一事件/状态机 → 进程与策略内核 → 真实 Run 闭环 → 审批/取消 → Artifact/验收 → 兼容回归 → Alpha`

任何一个适配器若只能“启动并显示日志”，但不能可靠处理终态、权限、取消和产物，则不得进入 Alpha 成功指标。

### 4.2 并行泳道

| 泳道 | W1-W2 | W3-W4 | W5-W6 | W7-W8 | W9-W10 |
|---|---|---|---|---|---|
| A 领域/客户端 | 工程骨架、领域状态机、SQLite spike | Workspace/Task/Run、2D 任务板 | 审批、交付、验收 UI | 恢复/通知、世界入口 | onboarding、诊断、打包 |
| B 适配器/系统 | 双 CLI 契约、取消/权限/恢复 spike | Process Supervisor、Codex/Claude Adapter v1 | 真实审批、产物与失败路径 | 兼容 fixtures、性能与降级 | 版本矩阵、安全回归 |
| C 3D/交互 | 体验预算与状态映射草图 | 不进入生产主线 | 固定场景技术原型 | 世界视图实现与 A/B | 性能优化、默认关闭策略验证 |
| D 产品/研究 | 访谈、工作流录屏、威胁建模 | 可用性走查 | 端到端任务测试 | 2D-only/2D+3D 对照 | 20-50 人邀请 Alpha |

## 5. 十周里程碑

### 5.1 M0：阻断风险验证（W1-W2）

目标：在建立完整 UI 前，确认双工具深适配和本地安全模型可行。

主要任务：

- DEV-001：建立 Tauri 2 + React + Rust workspace、格式化、静态检查和测试流水线。
- DEV-002：定义领域对象、Run 状态机、Event envelope 与错误码约定。
- DEV-003：定义 `Adapter` 最小契约和 capability manifest，显式包含审批模式。
- DEV-004：Codex spike：`exec --json`、实验性 app-server/exec-server、会话标识、权限行为、取消、恢复、退出码、产物。
- DEV-005：Claude spike：stream-json、权限请求、取消、恢复、hook、预算和产物。
- DEV-006：进程树治理 spike：优雅取消、超时、强制停止、残留进程检测。
- DEV-007：Workspace/Policy 威胁模型：路径越界、符号链接、命令注入、网络与 secret。
- DEV-008：SQLite 事件追加、崩溃重放与当前状态重建 spike。
- DEV-009：完成 6-10 名目标用户访谈和至少 4 条真实工作流录屏。
- DEV-010：形成 ADR-001 至 ADR-005。

退出标准：

- 两款 CLI 均能完成最小真实任务，并输出可解析的稳定事件或明确降级结果。
- Codex 审批路径已作出书面决策：逐动作协议、PTY，或 `sandbox_policy_only` 降级；能力差异可被 UI 和测试表达。
- 取消在 5 秒内进入 `cancelling`；10 秒无响应可标记 `forced_stop_required`。
- Run 崩溃后可从 SQLite 重建，不把未知终态伪装为完成。
- 路径越界、参数变化和危险命令能够在执行前被 Policy 拒绝或请求审批。
- 若任一深适配无法证明权限、取消或终态可控，召开 Pivot 评审，不进入 M1。

### 5.2 M1：控制骨架（W3-W4）

目标：形成可启动双工具只读/低风险任务的 2D 控制台。

主要任务：

- DEV-101：工具发现、版本/认证/能力检查与连接管理。
- DEV-102：Workspace 信任确认、目录规范化与 Policy 配置。
- DEV-103：Task 创建表单：目标、上下文、允许范围、验收、测试和回滚。
- DEV-104：Run 队列、最多两个并行槽位和手动执行器选择。
- DEV-105：Process Supervisor 与 Adapter Host 生产实现。
- DEV-106：规范事件写入、状态投影和 2D 任务板。
- DEV-107：日志脱敏和基础诊断页。
- DEV-108：第一批成功、无认证、版本不兼容、退出码异常 fixtures。

退出标准：

- 新环境 15 分钟内可连接两款工具、创建 Workspace 和首个任务。
- 两个隔离 Run 可以并行，状态不会串线。
- UI 重启后队列、运行记录和未知状态可以恢复。
- 未经信任的目录不能直接启动非交互 CLI。

### 5.3 M2：任务控制闭环（W5-W6）

目标：把“运行中”推进到可审批、可取消、可交付、可人工验收。

主要任务：

- DEV-201：Approval 模型、参数指纹、有效期、批准/拒绝/缩小范围，以及适配器审批模式差异展示。
- DEV-202：暂停/取消/恢复和残留进程处置 UI。
- DEV-203：Artifact 识别：文件、diff、测试结果、结构化完成报告。
- DEV-204：人工接受、退回修改、续跑与审计记录。
- DEV-205：错误分类与恢复建议，不以“未知错误”直接结束。
- DEV-206：端到端真实任务套件：连接 → 双任务 → 审批 → 取消/恢复 → 验收。
- DEV-207：审批绕过、参数漂移、符号链接竞态、日志泄密安全回归。

退出标准：

- `per_action` 适配器的所有高风险动作先产生绑定具体参数的审批请求；`sandbox_policy_only` 适配器若不能证明动作在冻结策略内安全，则必须阻止并要求以更窄策略重新启动，不得静默执行。
- 拒绝后同一动作不得执行；参数变化必须重新审批。
- Run `completed` 不等于 Task `accepted`，人工验收状态清晰可追溯。
- 真实 Codex 与 Claude 任务各完成至少 20 次端到端回归。
- 启动成功率达到 95%，终态一致率达到 99% 的预发布目标。

### 5.4 M3：最小世界视图（W7-W8）

目标：增加不干扰控制闭环的固定 3D 状态投影。

主要任务：

- DEV-301：固定低多边形办公室场景与资产预算。
- DEV-302：Idle、Planning、Running、Waiting Approval、Blocked、Reviewing、Completed、Sleeping 映射。
- DEV-303：角色/工位点击跳转到同一 Run 详情。
- DEV-304：3D 懒加载、卸载、低功耗和崩溃隔离。
- DEV-305：Tauri capability 限制，世界窗口不具备进程、写文件或凭据能力。
- DEV-306：2D/3D 事件回放一致性测试。
- DEV-307：2D-only 与 2D+3D 理解速度、点击和主动打开率测试。

退出标准：

- 关闭或卸载 3D 后，所有任务控制能力完整可用。
- 2D 与 3D 的同一 Run 状态无分叉；事件回放结果一致。
- 3D 不持有系统执行权限，渲染失败不影响后台 Run。
- 性能预算达到：空闲 CPU 低占用、无活动时降帧、内存上限由 W1 spike 后固化。

### 5.5 M4：邀请制 Alpha（W9-W10）

目标：完成兼容性、安全、恢复、打包和真实用户验证。

主要任务：

- DEV-401：Codex/Claude 版本兼容矩阵与自动 fixture 回归。
- DEV-402：崩溃、乱码、长输出、超时、磁盘满、数据库迁移和升级回滚测试。
- DEV-403：诊断包生成与 secret/路径脱敏检查（2026-07-17 已完成元数据白名单、二次扫描与原子导出验证）。
- DEV-404：macOS 签名、公证、安装、自动更新和卸载数据策略（2026-07-17 已完成仓库打包/签名公证 workflow、数据保留与降级保护；真实 Apple 证据、干净机生命周期和自动更新仍开放）。
- DEV-405：首次连接 onboarding、示例任务和错误恢复引导（2026-07-17 已完成派生式 5 步进度、受信任工作区内幂等只读示例、正常 Run/验收闭环与错误码恢复；真人首次使用证据归 DEV-407）。
- DEV-406：外部安全评审与高风险问题修复（2026-07-17 已完成内部预审、信任撤销、SQLite NOFOLLOW/私有权限、安全漂移 CI 门及外部评审包；独立评审、冻结 commit 与 Critical/High 复测仍开放）。
- DEV-407：20-50 名目标用户邀请、支持与指标看板（2026-07-18 已完成无身份、无内容、无路径、无自动上传的本地聚合看板与逐次同意导出，以及邀请/支持/口径操作手册；真实招募、支持记录、聚合结论与 Go/Pivot/No-Go 仍开放）。

退出标准：

- 兼容矩阵覆盖 Alpha 允许的 CLI 版本；未知版本默认降级或阻止。
- 发布包在干净 macOS 用户环境可安装、启动、升级和卸载。
- 诊断包不包含 token、prompt、项目名、完整路径或未脱敏日志。
- 关键安全问题清零；高风险问题未关闭不得发布。
- 达到 PRD Go 阈值，或形成明确 Pivot/No-Go 决策。

## 6. 最小适配器契约

每个深适配器至少实现：

```text
probe() -> ToolIdentity + Version + AuthState
capabilities(version) -> CapabilityManifest
  # 必含 approval_mode: per_action | sandbox_policy_only | none
prepare(task, workspace, policy) -> LaunchPlan
start(launchPlan) -> AdapterSession
sendInput(session, input)
streamEvents(session) -> NormalizedEvent*
respondApproval(session, approvalDecision)
cancel(session, deadline)
resume(sessionRef)
collectArtifacts(session) -> ArtifactCandidate*
reconcile(session) -> ObservedTerminalState
diagnostics(session) -> RedactedDiagnosticBundle
```

关键约束：

- 适配器不能自行扩大 Workspace/Policy。
- 审批模式必须进入 capability manifest 与 Run 详情；`sandbox_policy_only` 不得展示成逐动作审批。
- `LaunchPlan` 在执行前冻结并生成指纹；参数改变需重新过策略和审批。
- 解析失败进入 `degraded` 或 `uncertain`，不能猜测状态。
- 取消必须覆盖父进程、子进程、PTY 和残留检查。
- capability 以实际版本 fixture 证明为准，不以帮助文本直接宣称支持。

## 7. 测试策略

### 7.1 测试层级

- 单元：领域状态机、Policy、路径规范化、参数指纹、事件投影。
- 契约：两款适配器对同一 fixture 产生规范事件与错误码。
- 集成：真实 CLI、SQLite、进程树、取消/恢复、Artifact。
- 端到端：从连接到人工验收的完整用户路径。
- 安全：越界写入、命令/参数注入、审批绕过、token 泄露、符号链接竞态。
- 恢复：应用崩溃、Agent 崩溃、系统休眠、网络变化、磁盘满和升级失败。
- 性能：长日志、并行双 Run、3D 开关、事件回放与空闲资源占用。

### 7.2 必备 fixture

- 成功、失败退出码、无输出、非 UTF-8、超长单行、超长会话。
- 等待审批、拒绝审批、审批后参数变化、重复审批事件。
- 正常取消、忽略信号、派生子进程、僵尸/孤儿进程。
- 会话可恢复、会话丢失、CLI 版本不兼容、结构字段新增/缺失。
- 产物位于允许目录、越界目录、符号链接目标、文件删除/重命名。
- 日志包含假 token、路径、用户名、项目名时的脱敏。

## 8. Claude Code CLI 协作计划

Codex 负责核心架构、领域模型、权限语义、状态机、数据库迁移和最终合并审查。Claude Code CLI 适合独立处理：

- CC-01：整理每个 CLI 版本的 capability/fixture 矩阵草案。
- CC-02：为指定 Adapter 契约生成反例和故障注入清单。
- CC-03：独立审查 Threat Model 与审批绕过路径。
- CC-04：生成测试样例、mock 输出和长输出/乱码 fixtures。
- CC-05：审查某个独立 PR 的边界、错误处理和测试覆盖。
- CC-06：校对开发文档、迁移说明与 Alpha 排障手册。

协作规则：

- 默认只读分析；需要写入时限定明确目录和文件集合。
- 不让两个执行器同时修改同一核心文件。
- 支线输出必须包含假设、证据、未覆盖项和可复现命令。
- Codex 在进入主分支前重新运行测试并检查 diff，不把另一模型的结论视为事实。
- 涉及权限、凭据、迁移和发布的变更必须由 Codex 主审，并保留人工确认。

## 9. 首个迭代任务包（W1）

### Sprint Goal

证明工程骨架、事件模型和双 CLI 最小调用路径成立，并获得第一批真实工作流证据。

### 承诺任务

| ID | 任务 | 建议负责人 | 依赖 | 验收 |
|---|---|---|---|---|
| S1-01 | 初始化 Tauri/React/Rust monorepo 与 CI | 客户端 | 无 | 本地与 CI 均通过 lint/typecheck/unit/build |
| S1-02 | 定义领域对象和 Run 状态迁移测试 | Codex 主导 | 无 | 非法迁移被拒绝，状态可由事件重放 |
| S1-03 | 定义 Adapter 契约与 capability schema | Codex 主导 | S1-02 | 同一接口表达共性，`approval_mode` 等字段诚实暴露差异 |

### W1 结束演示

- 在开发机上从统一 spike runner 分别启动一个 Codex 和 Claude 只读任务。
- 两者事件被转换成同一最小 `Run/Event` 结构并写入 SQLite。
- 应用/runner 重启后可以重放并显示真实终态或 `uncertain`。
- 展示一次取消实验和一次越界策略拒绝，不要求完整 UI。

## 10. 项目治理

- 每日：关键路径阻塞、适配器 fixture 变化、安全问题同步。
- 每周：一场可运行演示，不以截图或 mock 状态代替真实 CLI。
- 每个里程碑：退出标准逐项签署；未通过项必须明确降级、延期或停止。
- 所有协议、状态、权限和数据库字段变更必须同步 schema、迁移、fixtures 和文档。
- 计划范围变化只通过 P0/P1/非目标评审进入，不在迭代中临时夹带。

## 11. Go / Pivot / No-Go 检查点

- W2：双 CLI 是否能被可靠控制，而不是只能被启动。
- W4：统一任务/事件/策略是否减少工具切换，还是增加一层配置负担。
- W6：审批、取消、Artifact 与人工验收闭环是否真实成立。
- W8：3D 是否提高状态理解或主动使用，而不是拖累主路径。
- W10：目标用户是否出现可重复的 Accepted Tasks/WAU 与留存信号。

任一阶段若核心证据不成立，应优先 Pivot 到更窄的本地双工具监督台，而不是用更多适配器、自动路由或 3D 功能掩盖闭环缺失。
